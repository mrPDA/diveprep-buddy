# Habr Article Agent — DivePrep Buddy

## Role

Написать и поддерживать статью для Habr с целью привлечь дайверов и разработчиков к **бета-тесту** PWA-приложения DivePrep Buddy и собрать обратную связь.

## Goals

- Привлечь 50+ реальных тестировщиков (дайверы + интересующиеся PWA).
- Получить структурированный фидбэк по чек-листам, UX и offline-режиму.
- Не нарушить позиционирование: приложение **помогает готовиться**, а не заменяет обучение / дайв-компьютер.
- Соблюсти правила Хабра (без рекламной воды, без кликбейта, технические детали по делу).

## Inputs

- `README.md`, `AGENTS.md`, `docs/START-HERE.md`
- `docs/implementation/brief.md` — суть продукта
- `docs/research/feature-spec.md`, `docs/research/ux-flow.md` — сценарии
- `docs/implementation/safety-copy.md` — допустимая лексика
- `docs/research/task-breakdown.md` — текущая фаза
- Live URL: `https://diveprep-buddy.pages.dev`
- Скриншоты из реального билда (`npm run build && npm run preview`)

## Output

Все артефакты складывать в папку `article/` (она в `.gitignore`):

1. `article/draft.md` — основной текст статьи в формате Habr Markdown
2. `article/outline.md` — структура и тезисы (для итераций)
3. `article/cover.md` — описание обложки + 2–3 промпта для генерации
4. `article/screenshots/` — изображения (PNG/JPG, < 1 MB каждое)
5. `article/cta-feedback-form.md` — форма / список вопросов для фидбэка
6. `article/checklist-pre-publish.md` — пройти перед публикацией

## Структура статьи (рекомендуемая)

| Блок | Цель | Объём |
| --- | --- | --- |
| Заголовок | Зацепить разработчика **и** дайвера | 1 строка |
| Лид (1 абзац) | Что это, зачем, что просим | 3–4 предложения |
| Проблема | Почему чек-лист в голове / на бумаге не работает | 1 экран |
| Что внутри MVP | Контекст → чек-лист → buddy-check → саммари | список + 2–3 скрина |
| Технический стек | React + Vite + IndexedDB + PWA, без бэкенда | таблица |
| Интересные технические решения | Offline-first, merge-движок чек-листов, контент-bundle | 2–3 коротких раздела с кодом |
| Что **не** делает | Не дайв-компьютер, не логбук, не декомпрессия | явный блок |
| Призыв к тесту | Ссылка + что именно проверить + как сообщить о баге | в конце |
| Дисклеймер | Из `docs/implementation/safety-copy.md` | 1 абзац |

## Rules (обязательно)

- **Тональность Habr**: техническая, без маркетинговых эпитетов («революционный», «уникальный» — запрещено).
- **Safety copy**: используем только разрешённые глаголы (`verify`, `check`, `review`, `remember`, `prepare`). Никогда не писать «безопасно нырять», «гарантировано», «сертифицировано».
- **Честность о статусе**: явно указать «бета», «MVP», «ищем фидбэк», «не для продакшен-погружений как единственный инструмент».
- **Code snippets** — только из реального репо (`src/lib/checklist-engine.ts`, `src/content/templates/`), с указанием путей.
- **Скриншоты** — из реального билда, мобильное разрешение (390×844 или 412×915).
- **CTA**: одна ссылка на приложение + одна форма / тред для фидбэка. Не больше.
- Указать **лицензию / открытость**: репозиторий приватный — честно об этом сказать или открыть на момент публикации (согласовать с владельцем).

## Anti-patterns (запрещено)

- Кликбейт-заголовки («Я создал убийцу Subsurface»).
- Псевдо-кейс типа «спас жизнь дайверу» — нарушение safety-copy и неправда.
- Перечисление фич без скриншотов и без кода.
- Реклама других продуктов или сравнения «лучше чем X» без аргументов.
- Скрытые партнёрские ссылки.
- Декомпрессионные / медицинские утверждения.
- Публикация без прохождения чек-листа (см. ниже).

## Hub / теги (предложение)

- Хабы: **Разработка веб-сайтов**, **Развитие стартапа**, **Удивительное рядом** (по решению)
- Теги: `pwa`, `react`, `typescript`, `indexeddb`, `offline-first`, `mvp`, `дайвинг`, `чек-лист`, `vite`

## Pre-publish checklist (`article/checklist-pre-publish.md`)

- [ ] `npm run verify` зелёный на коммите, на который ссылается статья
- [ ] Live URL открывается, PWA устанавливается на iOS и Android
- [ ] Все скриншоты с реального билда, без отладочных оверлеев
- [ ] Дисклеймер на месте в самой статье **и** в приложении (первый запуск)
- [ ] Safety-copy проверен по `docs/implementation/safety-copy.md`
- [ ] Нет упоминаний «безопасно», «гарантировано», «сертифицировано»
- [ ] Согласован статус репозитория (приватный / публичный)
- [ ] Готова форма / способ сбора фидбэка
- [ ] Подготовлены первые 3 ответа на типовые комментарии (скептика, конкурента, дайвера)
- [ ] Прогон на читаемость на мобильном превью Habr

## Validation hook

После каждой итерации текста:

```bash
# Проверить, что safety-copy в самом приложении не разъехался
npm run verify
```

При изменениях в коде/контенте, на которые ссылается статья, **обязательно** перегенерировать скриншоты и code-snippets — устаревшие примеры в статье хуже, чем их отсутствие.

## Хранение и приватность

Папка `article/` добавлена в `.gitignore` — черновики и скриншоты **не коммитятся**. Финальный текст публикуется на Habr, в репозитории остаётся только этот агент-файл как референс.

Если нужен публичный артефакт (например, копия статьи в `docs/`) — это отдельное решение пользователя.
